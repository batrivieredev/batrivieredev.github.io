TypeScript - Bonnes Pratiques

Guide des meilleures pratiques pour écrire du code TypeScript propre, maintenable et sécurisé.

Types et interfaces

✅ Bonnes pratiques

// Interfaces pour les objets
interface User {
  id: number;
  name: string;
  email: string;
}

// Types pour les unions et alias
type Status = 'pending' | 'approved' | 'rejected';
type UserId = number;

// Génériques contraints
interface Repository<T extends { id: number }> {
  findById(id: number): Promise<T>;
  save(item: T): Promise<T>;
}

// Types d'erreur personnalisés
class ValidationError extends Error {
  constructor(
    message: string,
    public field: string
  ) {
    super(message);
  }
}

❌ À éviter

// Éviter any
function process(data: any) { // ❌
  return data.someProperty;
}

// Éviter l'abus de types génériques
type GenericObject<T> = {
  [P in keyof T]: T[P] // ❌ Trop générique
};

Gestion des erreurs

// Types d'erreur personnalisés
interface AppErrorOptions {
  message: string;
  code?: string;
  status?: number;
}

class AppError extends Error {
  code: string;
  status: number;

  constructor(options: AppErrorOptions) {
    super(options.message);
    this.code = options.code ?? 'UNKNOWN_ERROR';
    this.status = options.status ?? 500;
  }
}

// Gestion des erreurs asynchrones
async function fetchData<T>(url: string): Promise<T> {
  try {
    const response = await fetch(url);

    if (!response.ok) {
      throw new AppError({
        message: `HTTP error ${response.status}`,
        code: 'API_ERROR',
        status: response.status
      });
    }

    return response.json();
  } catch (error) {
    if (error instanceof AppError) {
      throw error;
    }

    throw new AppError({
      message: 'Une erreur est survenue',
      code: 'UNKNOWN_ERROR'
    });
  }
}

Tests

// userService.test.ts
import { describe, it, expect } from 'vitest';
import { UserService } from './userService';
import type { User } from './types';

describe('UserService', () => {
  const service = new UserService();

  it('should create user with valid data', () => {
    const userData: Omit<User, 'id'> = {
      name: 'John',
      email: 'john@example.com'
    };

    const user = service.create(userData);

    expect(user).toMatchObject({
      ...userData,
      id: expect.any(Number)
    });
  });

  it('should throw on invalid email', () => {
    expect(() => {
      service.create({
        name: 'John',
        email: 'invalid-email'
      });
    }).toThrow('Invalid email');
  });
});

Organisation du code

// Barrel exports
// index.ts
export * from './user';
export * from './post';
export * from './comment';

// Types dans des fichiers séparés
// types.ts
export interface User { /* ... */ }
export interface Post { /* ... */ }

// Services dans des classes
// userService.ts
export class UserService {
  private users: User[] = [];

  async findById(id: number): Promise<User> {
    const user = this.users.find(u => u.id === id);
    if (!user) {
      throw new AppError({
        message: 'User not found',
        code: 'USER_NOT_FOUND',
        status: 404
      });
    }
    return user;
  }
}
Structure recommandée : 
src/
├── api/        # Appels API
├── components/ # Composants UI
├── hooks/      # Hooks personnalisés
├── services/   # Logique métier
├── types/      # Types et interfaces
└── utils/      # Utilitaires

Conventions de nommage

  • Interfaces : PascalCase, préfixe I optionnel (User ou IUser)
  • Types : PascalCase (UserRole, ApiResponse)
  • Énums : PascalCase (HttpStatus, UserRole)
  • Génériques : T, U, V ou descriptif (TEntity, TKey)
  • Fonctions : camelCase, verbe (getUserById, createPost)
